# ContextSwitcher

`ContextSwitcher` lets a person move between workspaces, organisations, or another application-defined context.

## Import

```tsx
import { Avatar, ContextSwitcher } from '@motech-development/breeze-ui';
```

## When to use

Use it when most application content is scoped to exactly one current context and context changes deserve a persistent shell control. Use `Select` for a form value, `Menu` for unrelated commands, or `NavigationList` when every option is a destination rather than a global context change.

The application owns the context collection, loading and error handling, authorisation, confirmation of destructive switches, persistence, and any route change after selection.

## Basic example

```tsx
import { Avatar, ContextSwitcher } from '@motech-development/breeze-ui';

<ContextSwitcher
  aria-label="Switch workspace"
  currentId="design"
  items={[
    {
      description: 'Primary workspace',
      icon: <Avatar initials="D" name="Design team" shape="square" size="md" />,
      id: 'design',
      name: 'Design team',
    },
    { id: 'research', name: 'Research team' },
  ]}
  manageLabel="Manage workspaces"
  onChange={(id) => setCurrentWorkspaceId(id)}
  onManage={() => navigate('/workspaces')}
  triggerLabel="Current workspace"
/>;
```

## State, items, and callbacks

This is a controlled pattern. `currentId` is a string id from `items`, or `null` when a selection is required; `onChange(id)` reports the requested next id and receives no DOM event. Update `currentId` after application validation succeeds. There is no `defaultCurrentId` or read-only mode. Disable unavailable items individually; when switching is wholly unavailable, render a non-interactive context summary.

Each `ContextSwitcherItem` requires a stable `id` and plain-string `name`. `name` supplies visible text and menu typeahead. `description` and `icon` add supporting context. `disabled` prevents that item from receiving selection or action. Item order is preserved.

When `currentId` is `null` or does not match an item, the trigger shows `emptyIcon` and `emptyName`, falling back to the `aria-label`. The optional management row appears only when both `manageLabel` and `onManage` are supplied. Its internal collision-safe key is not public and must not be referenced.

## Accessibility, keyboard, and internationalisation

`aria-label` names both trigger and menu; provide a concise translated action such as “Switch workspace”. `triggerLabel`, names, descriptions, empty copy, and management copy are application-owned and must be localised. Names should remain distinguishable without relying on icons, which should be decorative unless they convey information absent from the text.

The trigger opens with pointer activation, `Enter`, `Space`, or the platform menu shortcut. Arrow keys move through enabled items, typing moves by `name`, `Enter` or `Space` selects, `Home` and `End` move to boundaries, and `Escape` closes and restores trigger focus. Behaviour comes from the composed `Menu`. Logical alignment and `top start` placement follow provider direction.

The pattern does not fetch items or display loading, invalid, or error feedback. Show an appropriate `Skeleton`, `Spinner`, `Alert`, or `StatePanel` outside it, and avoid presenting an unexplained empty menu.

## Common mistakes and related components

Do not use array position as an id, pass a stale `currentId` silently, place an essential distinction only in colour, or perform the context change without updating controlled state.
