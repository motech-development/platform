# MetricCard

`MetricCard` presents one prominent, labelled metric without calculating, formatting, or fetching it.

## Import

```tsx
import { FormattedNumber, MetricCard } from '@motech-development/breeze-ui';
```

## When to use

Use it for a standalone key performance indicator, count, percentage, or other named value that deserves visual prominence.

## When not to use

Do not use it for a broad content summary, a set of equally weighted facts, bounded progress, or an inline value. Prefer `Card` for the broader summary, `DescriptionList` for several label-value facts, `Meter` for a value within a bounded range, and `FormattedNumber` for locale-aware inline output.

## Basic example

```tsx
import { FormattedNumber, MetricCard } from '@motech-development/breeze-ui';

<MetricCard
  description="Updated recently"
  label="Project progress"
  tone="subtle"
  value={<FormattedNumber options={{ style: 'percent' }} value={0.72} />}
/>;
```

## Value ownership, tones, and density

`label` and `value` are required React nodes; `description` is optional. Breeze applies tabular numerals to the value region but does not convert units, calculate trends, apply precision, or parse strings. Supply a domain-ready node and use `FormattedNumber`, `FormattedDateTime`, or `RelativeTime` when locale-aware display is appropriate.

| Prop value           | Effect                                                 |
| -------------------- | ------------------------------------------------------ |
| `tone="default"`     | Canonical light surface and strong label; the default. |
| `tone="subtle"`      | Quieter light surface for supporting page information. |
| `tone="inverse"`     | Dark shell-compatible surface with inverse copy.       |
| `density="regular"`  | Canonical internal content gap; the default.           |
| `density="spacious"` | More space between label, value, and description.      |

Tone and density are scalar presentation choices, not responsive objects. Choose tone from the surrounding surface and hierarchy, not from the numeric value's sign. Put semantic change or risk in explicit text rather than colour alone.

## Loading, empty, and error states

The application owns fetching and status. For a predictable loading layout, pass tone-compatible `Skeleton` nodes for label and value and hide the placeholder article with `aria-hidden`; announce loading in a separate status region. Do not pass an empty string as a finished value. Use an explicit localised placeholder when “not available” is meaningful, and use `StatePanel` or `Alert` when the entire metric cannot be loaded.

MetricCard has no controlled, uncontrolled, read-only, disabled, invalid, loading, or error prop and no callbacks.

## Accessibility and internationalisation

The semantic root is an `article` named by the visible label through a generated `aria-labelledby` relationship. Make labels concise and unique enough to distinguish adjacent metric articles. Include units in the formatted value or visible description, and do not rely on a visual abbreviation that a screen reader cannot interpret.

The card has no keyboard behaviour; interactive content generally does not belong in its value. Format numbers with the provider locale and dates/times with an explicit application time-zone policy. Translate label, description, and fallback copy. Logical layout supports both text directions, while numeral shaping follows the chosen locale and formatter options.

## Common mistakes and related components

Do not perform calculations in presentation copy, omit units, communicate good/bad state only through tone, put a form control inside the value, or render several unrelated measures in one card. Related components: [FormattedNumber](?path=/docs/formatting-formattednumber--docs), [Card](?path=/docs/data-display-card--docs), [DescriptionList](?path=/docs/data-display-descriptionlist--docs), [Meter](?path=/docs/feedback-meter--docs), and [StatePanel](?path=/docs/patterns-feedback-statepanel--docs).
