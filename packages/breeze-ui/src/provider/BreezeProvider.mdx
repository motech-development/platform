# BreezeProvider

`BreezeProvider` is the required environment boundary for every Breeze component. It establishes locale, reading direction, translated library messages, routing, portal placement, time-zone preferences, and the provider-scoped toast queue.

## Import

```tsx
import {
  BreezeProvider,
  type BreezeMessageOverrides,
  type BreezeRouterAdapter,
} from '@motech-development/breeze-ui';
```

Import the compiled component styles once at the application entry point. A greenfield application may opt into the reset before the component styles.

```tsx
import '@motech-development/breeze-ui/reset.css';
import '@motech-development/breeze-ui/styles.css';
```

## When to use

- Wrap each independently rendered Breeze application root once.
- Supply the user's BCP 47 locale and, where temporal values require it, their IANA time-zone name.
- Add a router adapter when local Breeze links should use client-side navigation.
- Override `messages` when an application localises Breeze-owned generic text and announcements.

## When not to use

- Do not nest providers merely to style a subsection; Breeze has one canonical theme and typed component variants.
- Do not use it as application state, authentication, data-fetching, or form context. Keep those concerns in application-owned providers.
- Do not mount a separate provider around each overlay or toast. The nearest application provider already owns their shared environment.

## Basic use

```tsx
import { BreezeProvider, Typography } from '@motech-development/breeze-ui';
import '@motech-development/breeze-ui/styles.css';

export function Application() {
  return (
    <BreezeProvider locale="en-GB" timeZone="Europe/London">
      <Typography as="h1">Account settings</Typography>
    </BreezeProvider>
  );
}
```

## Environment and behaviour

The provider renders a `data-breeze-root` element with `lang` and `dir`. When `direction` is omitted, Breeze infers `ltr` or `rtl` from `locale`; an explicit direction always wins. Use a BCP 47 locale such as `en-GB`, `fr-FR`, or `ar-SA`, not an application-specific language key.

When `portalContainer` is omitted, Breeze creates a provider-owned portal host in `document.body` and applies the same locale and direction to it. Pass a stable `HTMLElement` only when overlays must live in an application-owned host. Portal creation is guarded for server rendering.

The provider also owns the toast region. At most `toastLimit` toasts are visible; additional toasts queue. `toastLimit` must be a positive integer and throws a `RangeError` otherwise.

## Routing

The adapter is deliberately small:

```tsx
import {
  BreezeProvider,
  Link,
  type BreezeRouterAdapter,
} from '@motech-development/breeze-ui';

const router: BreezeRouterAdapter = {
  navigate(href) {
    window.history.pushState(null, '', href);
  },
};

export function RoutedApplication() {
  return (
    <BreezeProvider locale="en-GB" router={router}>
      <Link href="/preferences">Preferences</Link>
    </BreezeProvider>
  );
}
```

Link-capable components use `router.navigate(href)` for an unmodified primary activation of a local `/path`. External URLs, downloads, alternate targets, and modified clicks retain native browser behaviour. The application remains responsible for updating its rendered route after navigation.

## Messages

`messages` is a partial override. Unspecified entries retain the built-in domain-neutral `en-GB` text.

```tsx
import type { ReactNode } from 'react';
import {
  BreezeProvider,
  type BreezeMessageOverrides,
} from '@motech-development/breeze-ui';

const messages: BreezeMessageOverrides = {
  clear: 'Effacer',
  close: 'Fermer',
  loading: 'Chargement',
  noResults: 'Aucun résultat',
  notifications: 'Notifications',
};

export function LocalisedApplication({ children }: { children: ReactNode }) {
  return (
    <BreezeProvider locale="fr-FR" messages={messages}>
      {children}
    </BreezeProvider>
  );
}
```

The complete catalogue is `clear`, `close`, `fileCountRejection`, `fileSizeRejection`, `fileTypeRejection`, `hidePassword`, `loading`, `nextPage`, `noResults`, `notifications`, `page`, `previousPage`, `selectDate`, `selectDateTime`, `selectFiles`, `selectTime`, `showPassword`, and `unreadNotifications`. Translate meaning rather than control-specific wording, and keep accessible labels concise.

## Accessibility and keyboard interaction

The provider itself adds no keyboard commands. Its locale and direction feed the keyboard, focus, and announcement behaviour of descendant controls. Keep one consistent locale and direction for a coherent subtree, translate the message catalogue, and do not remove the provider-owned portal root from the document while an overlay is open.

Setting `direction="rtl"` affects logical layout and interaction direction without changing the language. Use an explicit override for testing or exceptional mixed-language surfaces; otherwise prefer locale inference.

## Common mistakes and limitations

- Rendering any Breeze component outside `BreezeProvider` throws an error.
- `locale` is required even when the application only uses visual primitives.
- `timeZone` must be an IANA name such as `Europe/London`, not an offset such as `+01:00`.
- Do not recreate built-in toast or portal infrastructure in each feature.
- Do not pass an element that is replaced on every render as `portalContainer`.
- Message overrides localise Breeze-owned generic UI only; application copy remains application-owned.

## Related components

- Use `Link` and `LinkButton` with `router` for router-neutral navigation.
- Use `FormattedDateTime`, `FormattedNumber`, `FormattedList`, and `RelativeTime` for provider-locale display formatting.
- Use `Toast` and `ToastRegion` for notifications owned by this provider boundary.
