# DropZone

`DropZone` accepts native files through drag and drop, validates them and returns successfully read files to the application.

## Import

```tsx
import { DropZone, FileTrigger } from '@motech-development/breeze-ui';
```

## When to use

Use `DropZone` when dragging files from the operating system is a useful primary interaction. Always provide a keyboard- and touch-operable `FileTrigger` fallback when selection is required.

## When not to use

Use `FileTrigger` alone for a compact chooser or the `FileUpload` pattern for a complete upload workflow. `DropZone` does not upload, store, preview or remove files.

## Basic example

```tsx
import { DropZone, FileTrigger } from '@motech-development/breeze-ui';

<DropZone.Root
  acceptedFileTypes={['application/pdf']}
  aria-label="Documents"
  onFiles={(files) => queueFiles(files)}
>
  <DropZone.Label>Drop documents here</DropZone.Label>
  <DropZone.Description>PDF documents only.</DropZone.Description>
  <FileTrigger.Root
    acceptedFileTypes={['application/pdf']}
    onFiles={queueFiles}
  >
    <FileTrigger.Trigger>Choose documents</FileTrigger.Trigger>
  </FileTrigger.Root>
  <DropZone.Feedback />
</DropZone.Root>;
```

## Anatomy

| Part                   | Purpose                                                                            |
| ---------------------- | ---------------------------------------------------------------------------------- |
| `DropZone.Root`        | Owns native drop interaction, file reading, validation state and callbacks.        |
| `DropZone.Label`       | Persistent visible drop-target label.                                              |
| `DropZone.Description` | Associated guidance about interaction or accepted files.                           |
| `DropZone.Feedback`    | Announces Breeze-owned file count, type and size rejections.                       |
| `DropZone.Error`       | Application validation message, rendered as an alert only while `invalid` is true. |

Keep all parts inside `Root`. A nested `FileTrigger` is a separate public compound component and should use the same acceptance rules to avoid inconsistent results.

## Files, failures and callbacks

`acceptedFileTypes` accepts exact MIME types, wildcard groups such as `'image/*'`, and extensions such as `'.csv'`. `maxFileSize` is bytes. `allowsMultiple` defaults to `false`. `onFiles` receives the accepted native `File[]`; `onReject` receives rejected files with one or more stable reasons: `'file-count'`, `'file-size'` and `'file-type'`. Mixed valid and invalid drops deliver both accepted files and rejection records.

Reading precedes validation. If any dropped item cannot be read, the entire mixed drop is atomic: `onFileReadError` runs once with `DropZoneFileReadFailure[]`, while `onFiles` and `onReject` are not called. Each failure supplies `cause`, zero-based `itemIndex`, and browser-supplied `name` and `type`. File-reading failures are distinct from constraint rejections and `Feedback` does not render them; show application-owned recovery guidance.

`invalid` controls `Error` and accessible invalid state. `required` is presentation and validation semantics only because a drop zone has no native submitted value. `disabled` prevents focus and dropping. There is no controlled file value, read-only state or loading prop; keep selected files and upload progress in application state.

## Keyboard, accessibility and direction

- The drop target can receive focus, but dragging is not a complete keyboard interaction. Include a clearly labelled `FileTrigger.Trigger` fallback.
- Provide a concise root accessible name, such as `aria-label="Documents"`, and a persistent visible `Label`. Use `Description` for type, size and count constraints.
- `Feedback` and `Error` use alert semantics. Do not render generic error prose in `Feedback`; it is reserved for Breeze constraint rejections.
- Active drop-target appearance is supplementary. Do not rely on border colour alone to communicate acceptance, rejection or required state.
- Visible content inherits provider locale and direction. Native filenames and files are not transformed; Breeze-owned rejection text is fixed component copy, so use `onReject` when the product needs different domain or localisation handling.

## Common mistakes and related components

Do not make drag and drop the only route, treat client-side constraints as security, confuse byte counts with megabytes, or call `onFiles` yourself after `onFileReadError`. Keep nested `FileTrigger` constraints aligned. Use `FileAttachment` to display a selected file and `FileUpload` for the composed workflow.
